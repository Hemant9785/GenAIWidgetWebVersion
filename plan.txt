PROJECT GOAL

Build a plugin-based GenUI widget generation platform.

The system should take a user query like:
"Create a 7 day weather widget for Bangalore with rain chance"

And produce a rendered JSON-based UI widget.

Weather is the first supported domain, but the architecture must be plugin-based so Finance, Calendar, News, Sports, etc. can be added later without changing the core orchestration.

Core pipeline:

User Query
→ LLM-1 Capability Router
→ Build LLM-2 tool set (default tools + domain tools)
→ LLM-2 Variable Planner (agentic tool-calling loop)
→ Backend Variable Resolver
→ LLM-3 Layout Generator
→ JSON UI Validator
→ Web Renderer

LLM Provider: OpenAI API (all LLM calls use OpenAI SDK).

TWO-TIER TOOL ARCHITECTURE

The system has two categories of tools:

1. DEFAULT TOOLS (always present in LLM-2 prompt)

These tools are always injected into LLM-2 regardless of domain.
They let LLM-2 dynamically explore plugin content.

Default tools:
- list_skills   → lists available skills (path + description)
- list_tools    → lists available tools (path + description)
- list_assets   → lists available assets (path + description)
- read_skill    → reads full skill content by path
- read_tool     → reads full tool definition by path

2. DOMAIN-SPECIFIC TOOLS (loaded after LLM-1 routing)

These are the actual API-calling tools from the selected plugin.
They are only available when LLM-1 routes to that domain.

Example domain tools for weather:
- /tool/weather/forecast
- /tool/weather/air-quality
- /tool/weather/historical
- /tool/location/geocode
- /tool/location/current-location

DEFAULT TOOL DEFINITIONS

list_skills:
  Parameters: { domain?: string }
  Returns: [{ path, description }]
  Example: list_skills() → all skills across all plugins
  Example: list_skills({ domain: "weather" }) → only weather skills

list_tools:
  Parameters: { domain?: string }
  Returns: [{ path, description }]
  Example: list_tools() → all tools across all plugins
  Example: list_tools({ domain: "weather" }) → only weather tools

list_assets:
  Parameters: { domain?: string }
  Returns: [{ path, description }]
  Example: list_assets() → all assets across all plugins
  Example: list_assets({ domain: "weather" }) → only weather assets

read_skill:
  Parameters: { path: string }
  Returns: skill content (string or array)
  Path semantics:
    /skill/weather/{skillName} → returns content of a specific skill
    /skill/weather             → returns all skills in the weather domain
  Example: read_skill({ path: "/skill/weather/forecast" }) → single skill content
  Example: read_skill({ path: "/skill/weather" }) → all weather skills as array:
    [
      { "name": "weather", "path": "/skill/weather/weather", "content": "..." },
      { "name": "location", "path": "/skill/weather/location", "content": "..." }
    ]

read_tool:
  Parameters: { path: string }
  Returns: tool definition (JSON schema)
  Path semantics:
    /tool/weather/{toolName}  → returns definition of a specific tool
    /tool/weather             → returns all tool definitions in the weather domain
  Example: read_tool({ path: "/tool/weather/forecast" }) → single tool schema
  Example: read_tool({ path: "/tool/weather" }) → all weather tool schemas as array

Everything should be replaceable:
- LLM provider
- Router
- Variable planner
- Tool executor
- Renderer
- UI schema format
- Domain plugins

Create a registry system that discovers available plugins.

The registry serves two purposes:
1. Provides compact summaries to LLM-1 (path + description only)
2. Acts as the backend for DEFAULT TOOLS used by LLM-2

Each plugin should expose:

manifest.json

Example:

{
  "domain": "weather",
  "skills": [
    {
      "path": "/skill/weather",
      "description": "Weather widgets: current weather, forecast, rain, humidity, wind, AQI, UV."
    }
  ],
  "tools": [
    {
      "path": "/tool/weather",
      "description": "Open-Meteo weather forecast, hourly, daily, AQI and historical APIs."
    }
  ],
  "assets": [
    {
      "path": "/asset/weather",
      "description": "Weather icons, rain icons, temperature icons and condition icons."
    }
  ]
}

Registry must implement DefaultToolHandler interface:

Path resolution logic:

  /skill/weather           → return all skills in weather domain
  /skill/weather/forecast  → return single "forecast" skill
  /tool/weather            → return all tools in weather domain
  /tool/weather/forecast   → return single "forecast" tool
  /asset/weather           → return all assets in weather domain
  /asset/weather/icons     → return single "icons" asset

Path format: /{type}/{domain}/{name?}
  type   = skill | tool | asset
  domain = weather | finance | etc.
  name   = optional specific resource name

Important:
The LLM should never see full plugin files in LLM-1.
LLM-1 only sees path + description.
LLM-2 can dynamically load full content via default tools.

PHASE 6: LLM-2 VARIABLE PLANNER

Implement LLM-2 as an AGENTIC TOOL-CALLING LOOP using OpenAI API.

LLM-2 is NOT a single-shot call. It runs in a loop with tool calls.
Use gpt-4o (configurable via LLM2_MODEL env var).

Tool set injected into LLM-2 prompt:

1. DEFAULT TOOLS (always present, every request):
   - list_skills({ domain?: string })
   - list_tools({ domain?: string })
   - list_assets({ domain?: string })
   - read_skill({ path: string })
   - read_tool({ path: string })

2. DOMAIN-SPECIFIC TOOLS (from LLM-1 selected paths):
   - Only the tools selected by LLM-1 routing
   - Example: /tool/weather/forecast, /tool/location/geocode

OpenAI call:
  model: LLM2_MODEL (default: gpt-4o)
  tools: [...defaultToolDefinitions, ...domainToolDefinitions]

Agentic loop:

  messages = [system_prompt, user_message]
  while true:
    response = openai.chat.completions.create(messages, tools)
    if response has tool_calls:
      for each tool_call:
        if tool_call is default tool:
          result = defaultToolHandler.execute(tool_call)
        else if tool_call is domain tool:
          result = domainToolExecutor.execute(tool_call)
        append tool_call and result to messages
      continue loop
    else:
      parse response as final variable plan
      break

Input context (in system prompt):
- user query
- domain from LLM-1
- global variables (if any)

LLM-2 decides on its own what to read:
- It may call read_skill("/skill/weather") to understand capabilities
- It may call read_tool("/tool/weather/forecast") to understand parameters
- It may call /tool/location/geocode to resolve ambiguous locations
- It outputs the final variable plan when done

Final output (last message, no tool calls):

{
  "status": "finish",
  "clarification_required": false,
  "variables": [
    {
      "variable_name": "dailyForecast",
      "variable_type": "array",
      "semantic_type": "daily_weather_forecast",
      "source": {
        "type": "tool",
        "tool_path": "/tool/weather/forecast",
        "parameters": {
          "latitude": "{{location.latitude}}",
          "longitude": "{{location.longitude}}",
          "forecast_days": 7,
          "daily": [
            "weather_code",
            "temperature_2m_max",
            "temperature_2m_min",
            "precipitation_probability_max"
          ]
        }
      },
      "importance": "required"
    }
  ],
  "assets": []
}

Safety:
- Max loop iterations: 10
- Max total tool calls: 20
- Timeout per loop iteration: 30 seconds

Important:
Default tools let LLM-2 explore. Domain tools let LLM-2 resolve ambiguity.
Prefer finish with tool-backed variable sources over calling domain tools directly.
Do not call domain tools just to fetch normal data that the resolver can handle.

Expected outcome:

[ ] LLM-2 can call default tools to explore skills/tools
[ ] LLM-2 can call domain tools for disambiguation
[ ] Agentic loop runs until finish
[ ] Produces variable plans
[ ] Avoids unnecessary tool calls
[ ] Validates tool paths
[ ] Loop terminates within safety limits

  Dispatches default tool calls to the registry (DefaultToolHandler).

  list_skills  → registry.listSkills(domain)
  list_tools   → registry.listTools(domain)
  list_assets  → registry.listAssets(domain)
  read_skill   → registry.readSkill(path)

  - Input: name, count, language
  - Output: latitude, longitude, timezone, country, name

  /tool/weather/forecast
  - Uses Open-Meteo Forecast API
  - Supports current, hourly, daily
  - Supports forecast_days
  - Supports weather_code, temperature, rain, wind, humidity, UV

  /tool/weather/air-quality
  - Uses Open-Meteo Air Quality API

  /tool/weather/historical
  - Uses Open-Meteo Historical API

  /tool/location/current-location
  - For now mock with configured default location
  - Later replace with device location

Tool dispatch logic:
  if tool_name in [list_skills, list_tools, list_assets, read_skill, read_tool]:
    → route to DefaultToolHandler (registry)
  else:
    → route to DomainToolExecutor (API calls)

(e.g., geocode for location disambiguation). The resolver must handle:

- Execute domain tools for unresolved variables

- Store resolved variables
- Return clean variable map to LLM-3
